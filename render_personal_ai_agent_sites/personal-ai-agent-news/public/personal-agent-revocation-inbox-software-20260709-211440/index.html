Three workflows define the category.

The strongest products make memory decisions actionable instead of merely visible.

Review receipts that need a human decision.

The inbox should prioritize uncertain parses, sensitive reuse, stale memory, and cases where the user asked why.

Repair memory scope without losing history.

Users should roll back a deletion, narrow a memory further, renew an expired fact, or resend source evidence.

Search what changed across future behavior.

Search should answer "what did I revoke," "what action did this block," and "why did the agent stop suggesting this?"

Buyer checklist.

Use this checklist when evaluating personal agent revocation inbox software.

Receipt-first UI

Cards should summarize memory, source, reply, policy delta, and repair action.

Source evidence

Every card should point to the message, file, page, or user reply that created memory.

Plain reply verbs

Support keep, narrow, expire, delete, renew, and ask why.

Action impact

Show which future workflows or agent actions changed because of the receipt.

Search and filters

Filter by source, workflow, date, sensitivity, state, and channel.

Repair workflow

Let users undo, narrow further, resend evidence, or restore prior scope.

FAQ for revocation inbox software.

The category matters when memory control becomes a recurring personal agent workflow.

Is this just a settings page?

No. A settings page changes preferences. A revocation inbox keeps searchable receipts and shows future behavior changes after memory decisions.

Why not just use audit logs?

Audit logs are event-first. A revocation inbox is user-decision-first and should be legible to the person whose memory is being used.

Where does Super fit?

Super can provide the phone-native review surface that creates many of the inbox receipts.

What should teams measure?

Track needs-review receipts, failed policy deltas, missing source evidence, rollback rate, and blocked actions caused by revocations.
