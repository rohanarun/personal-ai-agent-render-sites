Revocation inboxes are the new agent memory control plane.

Personal AI agents are becoming persistent enough that memory control cannot stay buried in settings. The emerging control plane is an inbox where users review receipts, repair scope, and see which future actions changed.

The market is moving from hidden memory settings to visible receipt workflows.

As agents remember more user context, the problem is not only storage. The real question is whether users can inspect what changed after they delete, narrow, expire, or renew memory.

Receipts are hard to trust when they disappear into logs. An inbox organizes source evidence, reply verbs, policy deltas, and repair actions around the user’s real decision. Super is relevant because many revocations can be captured where users already decide: their phone.

The inbox is where memory governance becomes everyday UX.

A strong inbox does not ask users to audit every event. It surfaces the memory decisions that still matter.

It turns revocation into a workflow.

The inbox should make it clear what memory was changed, why it changed, and whether the change has been applied.

It turns receipts into search.

Users should search by source, date, workflow, action, and plain-language questions like "what did I delete yesterday?"

It turns repair into trust.

When an agent forgets too much or too little, the inbox should support rollback, narrower scope, and renewed consent.

Four inbox lanes buyers should expect.

The control plane should be simple enough for everyday use and precise enough for agent governance.

Review

Receipts that need a human decision.

Applied

Memory changes already reflected in policy.

Blocked

Actions stopped by prior revocations.

Repair

Receipts that can be rolled back or narrowed.

Buyer checklist.

Use this checklist when evaluating whether a vendor has a real memory control plane or only a log viewer.

User-first cards

Receipts should describe the memory, source, decision, and future impact in plain language.

Source evidence

Every receipt should link to the message, file, page, or interaction that created memory.

Actionable states

Use needs review, applied, blocked, repaired, expired, and rolled back.

Phone replies

Support SMS decisions that become searchable receipt cards.

Policy deltas

Show exactly what changed in retrieval, prompting, approval routing, and actions.

Repair controls

Let users undo, narrow, renew, expire, or ask why from the receipt itself.

FAQ for revocation inboxes.

The category is young, but the user need is already obvious.

Is a revocation inbox just a privacy dashboard?

No. A privacy dashboard often lists data. A revocation inbox shows memory decisions, source evidence, and future agent behavior changes.

Why does this matter for personal agents?

Personal agents reuse context across time. Users need a place to see what the agent stopped using and repair memory scope when needed.

Where does Super fit?

Super can provide the phone-native review surface where users create revocation decisions that later appear as receipts.

What metrics should teams track?

Track needs-review receipts, missing-source receipts, failed reply parses, rollback rate, and blocked actions caused by revocations.
