Core architecture.

A credible broker controls the full path from discovery through execution, evidence, memory, and deletion.

Secret registry

Store credentials and private values with owner, category, provenance, allowed uses, rotation state, and recovery policy. The registry exposes metadata and stable placeholders, not raw values.

Control: encrypted inventory

Grant engine

Evaluate user identity, tool identity, origin, browser session, field category, purpose, amount, recipient, approval state, risk score, and expiration before issuing access.

Control: policy-bound token

Approval router

Send a redacted decision packet through the user’s chosen channel. Bind the reply to one request and invalidate it when the page, action, amount, or destination changes.

Control: explicit user intent

Execution adapter

Resolve the token inside a trusted browser or API tool, place the value into the authorized field, prevent model-side echo, and expire access immediately after completion.

Control: least-privilege use

Evidence redactor

Replace values with stable placeholders across screenshots, OCR, DOM snapshots, action logs, videos, traces, support exports, and audit receipts.

Control: reviewable proof

Memory firewall

Prevent execution values and derived sensitive facts from entering summaries, vectors, profiles, learned rules, or future prompt context without separate policy and consent.

Control: non-persistent secrets

Four broker decisions.

Hover across the outcomes to see how the broker responds at execution time.

Release once

Resolve for one exact action, then destroy the execution grant.

Release scope

Allow a defined origin or workflow until a clear limit or expiration.

Challenge

Require stronger identity, clarification, edit, or renewed approval.

Deny

Block mismatched origin, purpose, amount, identity, policy, or deadline.

The broker should let an agent complete the task while making the raw secret unavailable to planning, conversation, replay, and memory.

Buyer checklist.

Use these requirements when evaluating secret-broker software for personal AI agents.

Model isolation

Raw values never appear in prompts, transcripts, model traces, evaluation logs, or model-side storage.

Origin binding

Grants are limited to exact HTTPS origins, applications, fields, actions, sessions, and tool identities.

Intent binding

Approval records purpose, consequence, amount, recipient, scope, expiration, and the user’s explicit reply.

Replay protection

Screenshots, OCR, DOM, videos, support exports, and audit logs use stable redacted placeholders.

Memory firewall

Temporary secrets and derived sensitive facts cannot become agent memory without separate policy.

Publishing boundary

When work feeds an AI website-building workflow, placeholders survive into drafts, forms, metadata, analytics, and deployment logs.

FAQ.

Secret brokers combine vaulting, policy, identity, approval, execution, evidence, and memory controls.

How is a secret broker different from a password manager?

A password manager stores and autofills credentials for a person. An agent secret broker issues purpose-bound, tool-bound, time-bound grants and produces evidence for autonomous execution.

Does the model need the raw value to fill a form?

No. A trusted execution adapter can resolve a placeholder and place the raw value directly into the browser field without returning it to the planning model.

Can a broker support payment methods?

Yes, when grants are bound to merchant origin, amount, recipient, session, approval, and short expiration, with appropriate payment and identity controls.

What should happen after execution?

Expire the token, redact evidence, record the result, prevent memory promotion, and provide rollback or repair when the action fails.
